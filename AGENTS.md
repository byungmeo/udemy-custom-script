# AGENTS.md

## 목적
- 이 저장소는 Udemy 강의 페이지에서 대본 스크립트를 추출하고, 커스텀 포맷으로 가공한 뒤, 강의 플레이어 위에서 실시간으로 커스텀 자막을 출력하는 Chrome Extension 프로젝트다.
- 현재 저장소에는 Chrome Extension 튜토리얼 템플릿 흔적이 남아 있을 수 있다. 기능 구현이나 리팩터링 시 템플릿 예제보다 이 문서의 제품 요구사항을 우선 기준으로 삼는다.

## 현재 프로젝트 상태
- `manifest.json`, `popup.html`, `README.md`는 아직 초기 예제 수준일 수 있다.
- 샘플 원본 대본은 `sample_data/udemey_script_sample.vtt`를 기준 예시로 사용한다.
- 메타데이터 스키마 초안은 `schemas/udemy-custom-script-metadata.schema.json`에 둔다.
- 전체 텍스트 포맷 명세는 `docs/custom-script-format.md`를 기준으로 삼는다.
- 새 기능을 만들 때는 "Udemy 대본 추출/가공/출력" 목적과 직접 연결되는지 먼저 확인한다.

## 제품 요약
- 확장은 Udemy 강의의 타임라인 포함 대본을 추출한다.
- 추출한 대본을 메타데이터와 함께 자체 포맷으로 감싼다.
- 사용자는 확장에서 제공하는 복사 기능으로 이 포맷을 가져간 뒤, 외부 AI 또는 수동 작업으로 번역문을 채운다.
- 사용자는 완성된 커스텀 스크립트를 확장에 업로드하거나 붙여넣는다.
- 강의 플레이어 하단 컨트롤 패널에 추가된 확장 UI를 통해 표시 방식과 출력 순서를 커스터마이징한다.
- 강의 재생 중에는 현재 시점에 맞는 자막이 확장에서 관리하는 스크립트 기준으로 출력된다.

## 이 확장의 핵심 차별점
- 단순 추출 도구가 아니라, 플레이어 안에서 커스터마이징된 대본을 실시간 출력한다.
- Google Translate, DeepL 같은 특정 번역 도구에 종속되지 않는다.
- 사용자가 GPT, Gemini 등 외부 LLM과 원하는 번역 지침을 활용해 직접 번역 품질과 스타일을 통제할 수 있다.

## 향후 확장 방향
- 추후 외부 LLM과 API Key를 통해 직접 연동할 가능성을 염두에 두고 설계한다.
- 번역 파이프라인은 특정 제공자에 종속되지 않도록 분리한다.
- 프롬프트 생성, 번역 실행, 결과 저장, 자막 렌더링은 결합하지 말고 모듈 경계를 분명히 유지한다.
- 브라우저 종속적인 저장 API 위에 직접 비즈니스 로직을 얹지 말고, 저장소 어댑터 계층을 둬서 향후 외부 동기화 백엔드나 클라우드 저장소로 확장 가능하게 유지한다.

## 핵심 사용자 플로우
1. Udemy 강의 페이지에서 원본 대본을 추출한다.
2. 확장의 "클립보드에 복사" 기능으로 커스텀 포맷 스크립트를 복사한다.
3. 사용자가 외부 AI 또는 수동 편집으로 번역문을 채운다.
4. 사용자가 완성된 스크립트를 확장에 업로드하거나 붙여넣는다.
5. 확장 UI에서 번역문만 표시할지, 원문과 함께 표시할지, 출력 순서를 어떻게 할지 설정한다.
6. 강의 재생 시 타임라인에 맞춰 커스텀 자막이 출력된다.

## 커스텀 대본 포맷 요구사항

### 메타데이터 필수 항목
- 고유한 강의를 식별할 수 있는 정보
- 강의 정보
  - 예: 스킬 수준, 기본 언어, 자막 유무
- 현재 대본의 언어
- 대본의 최종 업데이트 날짜
- 전체 강의 제목
- 전체 강의 설명
- 현재 대본이 속한 Section 정보
  - 예: `Section 1: Introduction`
- 현재 대본이 속한 Lecture 정보
  - 예: `What is Common UI?`

### 메타데이터 직렬화 형식
- 메타데이터는 `JSON`으로 직렬화한다.
- 이유는 다음과 같다.
  - 문법이 더 엄격해서 파싱과 유효성 검사가 단순하다.
  - JavaScript 기반 Chrome Extension 환경에서 직렬화/역직렬화와 스키마 검증이 자연스럽다.
  - LLM이 구조를 안정적으로 유지하기 좋고, 프로그램이 후처리하기도 쉽다.
  - 향후 외부 API, 백엔드, 저장소 어댑터와 연결할 때 범용성이 높다.
- 메타데이터 블록은 명확한 구분자를 사용해 본문과 분리한다.
- 권장 구분자는 `<<<UDEMY_CUSTOM_SCRIPT_METADATA>>>` 와 `<<<END_METADATA>>>` 이다.
- 키 이름은 영어 `lower_snake_case`로 고정한다.
- 날짜/시각 필드는 파싱 안정성을 위해 `ISO 8601` 문자열로 저장한다.
- 비어 있는 선택 필드도 키 자체는 유지하고 `null` 또는 빈 문자열로 명시한다.
- 메타데이터 구조는 너무 깊게 중첩하지 말고, 최대 2단계 정도의 읽기 쉬운 구조를 유지한다.
- 본문은 사람이 그대로 읽고 편집할 수 있는 평문 블록 구조를 유지한다.
- `section_index`, `lecture_index`는 신뢰 가능한 값을 확보하지 못하면 `0`을 허용한다.
- `course.level`은 기계 처리용 안정 코드로 저장하고, 표시용 문자열은 `course.level_label`에 별도로 둔다.
- `course.description`은 가능하면 문단 경계를 유지하고, JSON 문자열 안에서는 `\n\n`로 문단 구분을 표현한다.

### 권장 포맷 예시
```text
<<<UDEMY_CUSTOM_SCRIPT_METADATA>>>
{
  "format_version": 1,
  "provider": "udemy",
  "identity": {
    "lookup_key": "udemy__course-12345__section-1__lecture-67890__lang-en",
    "course_id": "12345",
    "course_slug": "example-course",
    "section_id": "111",
    "section_index": 1,
    "lecture_id": "67890",
    "lecture_index": 3,
    "lecture_slug": "what-is-common-ui",
    "transcript_language": "en"
  },
  "course": {
    "title": "Example Course",
    "description": "First paragraph.\\n\\nSecond paragraph.",
    "level": "beginner",
    "level_label": "Beginner",
    "default_language": "en",
    "caption_available": true
  },
  "transcript": {
    "language": "en",
    "updated_at": "2026-03-11T21:00:00+09:00"
  },
  "section": {
    "title": "Section 1: Introduction"
  },
  "lecture": {
    "title": "What is Common UI?"
  }
}
<<<END_METADATA>>>
00:05.440 --> 00:06.160
Welcome.
환영합니다.
```

### 본문 구조
- 메타데이터 이후에는 대본 전체가 이어져야 한다.
- 각 자막 블록의 기본 형식은 아래 순서를 따른다.

```text
{타임라인}
{원문}
{번역문}
```

- 자막 블록 사이에는 빈 줄 하나를 둔다.
- 타임라인은 원본 대본의 시간 정보를 그대로 유지하는 것을 원칙으로 한다.
- 원문은 추출한 원본 텍스트를 최대한 손상 없이 보존한다.
- 원문 또는 번역문에 내부 줄바꿈이 있으면 export 시 공백으로 정규화해 각 필드가 한 줄을 유지하도록 한다.
- 번역문이 아직 없는 상태를 지원해야 한다면, 블록 구조를 깨지 말고 번역문 줄만 비워 두는 방향을 우선 고려한다.
- import/export 파일 포맷은 UTF-8 평문을 기본으로 하며, 파일 확장자를 하나로 고정해야 한다면 `.txt` 또는 `.md`를 사용할 수 있다.
- Markdown 생태계와 함께 다루기 쉽다는 점 때문에 기본값은 `.md`를 우선 고려하되, 실제 포맷의 표준은 파일 확장자가 아니라 "상단 JSON 메타데이터 블록 + 본문 자막 블록" 구조다.

## 구현 원칙

### 데이터 모델
- 최소한 다음 개념을 분리해서 다룬다.
  - Course
  - Section
  - Lecture
  - Transcript
  - TranscriptCue
- 가능하면 `TranscriptIdentity` 또는 이에 준하는 구조를 별도 개념으로 둔다.
- 강의를 식별하는 키와 현재 강의 내 Section/Lecture 위치 정보는 분리해서 저장한다.
- 강의 기본 언어와 현재 대본 언어는 별도 필드로 유지한다.
- 식별 정보는 단일 ID 하나만 저장하지 말고, 가능한 식별자를 묶은 `identity bundle` 형태로 유지한다.
- 가능한 경우 다음 항목을 함께 수집한다.
  - provider
  - course_id
  - course_slug
  - section_id
  - section_index
  - lecture_id
  - lecture_index
  - lecture_slug
  - transcript_language
- 내부 조회용 키는 위 식별자 묶음으로부터 결정적으로 생성한 `lookup_key`를 사용한다.

### 모듈 경계
- 추출 로직과 렌더링 로직을 분리한다.
- 포맷 직렬화/역직렬화 로직을 별도 모듈로 유지한다.
- 저장소 계층과 UI 계층을 직접 결합하지 않는다.
- 향후 LLM 연동을 고려해 번역 제공자 인터페이스는 교체 가능하게 설계한다.

### UI/UX
- 확장 UI는 Udemy 플레이어 기본 동작을 방해하지 않아야 한다.
- 사용자가 다음 옵션을 쉽게 바꿀 수 있어야 한다.
  - 번역문만 표시
  - 원문 + 번역문 동시 표시
  - 원문 우선 / 번역문 우선 출력 순서
- 플레이어에 주입하는 컨트롤은 기존 Udemy UI 변경에 취약할 수 있으므로, DOM 선택자와 마운트 전략은 가능한 한 방어적으로 작성한다.

### 포맷 안정성
- 커스텀 대본 포맷은 사람이 읽을 수 있으면서도 기계적으로 안정적으로 파싱 가능해야 한다.
- 포맷이 바뀌면 버전 필드 또는 호환 전략을 같이 정의한다.
- 메타데이터 키 이름, 블록 구분 규칙, 줄바꿈 규칙은 구현마다 달라지지 않도록 일관되게 유지한다.
- LLM이 수정하기 좋은 형식과 프로그램이 안정적으로 파싱하기 좋은 형식이 충돌할 경우, 메타데이터는 엄격한 JSON, 자막 본문은 고정 3줄 블록이라는 현재 규칙을 우선 유지한다.

### 식별자와 파일명 규칙
- 메타데이터 내부 `identity` 객체가 원본 진실(source of truth)이다.
- 파일명은 빠른 탐색과 유효성 검사를 위한 보조 수단으로 사용한다.
- 매칭 정확도는 제목보다 안정적인 ID를 우선 기준으로 삼는다.
- 파일명은 `lookup_key`와 동일한 구성 원칙을 따르는 결정적 문자열로 생성한다.
- 권장 규칙은 아래와 같다.

```text
udemy__course-{course_id}__section-{section_index}__lecture-{lecture_id}__lang-{transcript_language}.md
```

- `course_id`, `lecture_id`처럼 안정적인 값이 있으면 반드시 포함한다.
- `slug`나 제목 기반 문자열은 가독성을 위해 보조적으로 넣을 수 있지만, 파일명 매칭의 1차 기준으로 삼지 않는다.
- 일부 식별자를 확보하지 못한 경우에도 필드 순서는 바꾸지 말고, 확보한 값만 결정적으로 채운다.

### 저장 전략
- 저장소 계층은 브라우저 구현체와 분리된 추상화로 설계한다.
- 현재 기준의 기본 전략은 다음과 같다.
  - 사용자 설정, 최근 사용 강의, `lookup_key` 인덱스, 해시, 요약 메타데이터는 `WebExtensions storage.sync` 호환 계층에 저장
  - 전체 대본 본문과 대용량 데이터는 `IndexedDB` 또는 이에 준하는 로컬 구조화 저장소에 저장
- 이유:
  - 동기화 저장소는 여러 머신에서 같은 브라우저 계정으로 이어서 쓰기 좋다.
  - 하지만 전체 대본 본문을 장기적으로 전부 동기화 저장소에만 의존하는 것은 용량과 안정성 측면에서 부적합할 수 있다.
- 따라서 전체 스크립트의 표준 원본은 언제든 import/export 가능한 텍스트 패키지여야 한다.
- 가능하면 메타데이터 인덱스만으로도 "현재 코스/섹션/lecture에 해당하는 스크립트가 있는지" 빠르게 판별할 수 있어야 한다.
- 장기적으로 진짜 의미의 브라우저 비종속 다중 기기 동기화가 필요하다면, 별도 백엔드 또는 사용자 클라우드 저장소 어댑터를 추가하는 방향을 우선 고려한다.

## 변경 시 우선순위
- 1순위: 자막 동기화 정확성
- 2순위: 메타데이터 보존 정확성
- 3순위: import/export 포맷 안정성
- 4순위: 플레이어 UI와의 공존성
- 5순위: 향후 자동 번역/LLM 연동 확장성

## 검증 체크리스트
- 샘플 VTT에서 타임라인과 원문이 정확히 추출되는지 확인한다.
- 커스텀 포맷으로 내보낸 뒤 다시 불러와도 정보 손실이 없는지 확인한다.
- JSON 메타데이터를 LLM이 수정한 뒤 다시 가져와도 파싱이 안정적으로 되는지 확인한다.
- `lookup_key` 생성 규칙이 동일 입력에 대해 항상 같은 결과를 내는지 확인한다.
- Section/Lecture 매핑이 올바른지 확인한다.
- 번역문만 표시, 원문+번역문 표시, 출력 순서 변경이 정상 동작하는지 확인한다.
- 재생 시간 변화에 맞춰 기대한 자막 블록이 출력되는지 확인한다.
- 번역문이 비어 있는 경우에도 UI나 파서가 깨지지 않는지 확인한다.
- 동기화 가능한 인덱스와 로컬 대본 본문 사이의 참조가 깨졌을 때 복구 전략이 있는지 확인한다.

## 작업 지침
- 새 기능을 추가할 때는 아래 순서를 우선 고려한다.
  1. Udemy 페이지에서 어떤 데이터를 안정적으로 얻을 수 있는지 확인
  2. 내부 표준 데이터 모델로 정규화
  3. 커스텀 포맷으로 직렬화/역직렬화
  4. 저장 및 불러오기
  5. 플레이어 렌더링과 사용자 설정 반영
- 포맷이 애매한 부분은 임의로 여러 스타일을 허용하기보다, 우선 하나의 결정적 규칙을 정하고 유지한다.
- 사용자 데이터 손실 가능성이 있는 변경은 특히 신중히 다룬다.
- 특정 번역 서비스 전용 전처리/후처리 규칙은 코어 로직에 하드코딩하지 않는다.

## 협업 시 유의사항
- 요구사항이 명확하지 않으면 다음 항목을 우선 확인한다.
  - 메타데이터의 정확한 직렬화 형식
  - 강의 식별자 소스
  - import/export 저장 위치
  - 플레이어 DOM 주입 방식
- 브라우저 동기화만으로는 완전한 비종속 다중 기기 동기화가 어렵다는 점을 염두에 두고, 코어 데이터 모델은 항상 외부 저장소로 이전 가능하게 유지한다.
- README와 실제 제품 방향이 다를 수 있으므로, 문서 수정 시 이 문서와 제품 요구사항에 맞춰 함께 정리한다.
